Introduction

Welcome to the development log for my Game Boy emulator project! This project is a journey into understanding how the classic Game Boy hardware works by building a cycle-accurate emulator in C++.

Project Goals

  • Build a fully functional Game Boy emulator from scratch
  • Learn about CPU architecture, memory management, and hardware emulation
  • Document the development process for others to learn from

What You Need to Write an Emulator

The topics related to emulation can go quite deep, but to start with simple systems all you really need is some basic computer architecture knowledge, documentation and a somewhat fast programming language.

Computer Architecture

I will probably cover some things when absolutely necessary, but the gist of it is knowing a bit of how a CPU works. Things like how registers store data, how instructions are fetched and executed, and how memory is addressed.

Documentation

The best documentation on Game Boy is the pandocs. If you get stuck, looking at other emulator implementations can help you understand different approaches to solving problems.

Programming Language

The older the system is the more leeway you have with the programming language speed. I also prefer programming languages that allow you to use unsigned integers as that better represents what the hardware is capable of.

Let's say you have a 16-bit address space, this would mean addresses from 0 to 65,535, which is the maximum value of an unsigned 16-bit integer, as opposed to a range of -32,768 to 32,767. You can just use a 32-bit integer to access the values, but I think it's a bit easier with unsigned integers.

For this project I will be using C++, with a disclaimer that I'm not a C++ programmer and my code might end up not being idiomatic. Plain old C is a perfect language for this kind of project but I want to learn C++ at some point as most modern emulators are written in it, so I might as well use it.

Current Status

The emulator currently features a CPU with many core instructions implemented, complete PPU with all three rendering layers (background, window, sprites), and dual input support (keyboard + gamepad). It successfully runs games like Tetris for over 6 million cycles before hitting unimplemented instructions.

What's Working

  • ✓ Disassembler with full CB-prefixed instruction support
  • ✓ CPU execution core with cycle-accurate timing
  • ✓ Memory mapping with I/O registers
  • ✓ Many Sharp LR35902 instructions (load, arithmetic, logic, jumps, etc.)
  • ✓ PPU with background scrolling (SCX/SCY)
  • ✓ Window layer with proper line counter
  • ✓ Sprite rendering (8x8 and 8x16 modes, flipping, transparency, priority)
  • ✓ OAM scanning with 10-sprite-per-scanline limit
  • ✓ V-Blank and LCD STAT interrupts
  • ✓ Joypad input via keyboard (Z/X/Arrows/Shift/Enter)
  • ✓ Gamepad support (Xbox, PlayStation, Nintendo Pro controllers)
  • ✓ Hot-plugging for controllers
